# 商业化导购 Agent 总体路线图

## 最终目标

把当前导购聊天系统演进为一个可运营、可测试、可回归、可发布的商业化导购 Agent 产品：

- 聊天体验稳定，像资深导购协同工作，而不是返回僵硬模板
- 业务结果可信，客户、商品、任务和后链路都基于真实实体数据
- 生成式表达受控，允许自然生成，但不能改变业务事实
- UI 协议可演进，复杂结果以复合语义模块返回，并有兼容和降级策略
- 记忆、学习、技能、权限、审计、观测和发布流程都能持续运转

## 阶段性目标

### 基线版本 `v0.1`

- 建立工程化文档、分支流转、发布门禁和问题记录机制
- 固定 `develop -> test -> main` 流程
- 把现有仓库推到远端，形成后续迭代基线

### Version `1.0` 去写死与工程底座

- 去掉关键业务硬编码
- 拆清决策分层
- 建立结果稳定性契约
- 把测试从“固定答案断言”改成“事实稳定 + 语义可变”

### Version `1.1` 测试体系升级与语义评测

- 建立稳定性测试、生成变化测试、重复提问测试、DB 对照测试
- 引入语义评测和失败样本回流

### Version `1.2` 受控生成式 UI

- 从固定组件拼接升级为复合语义模块
- 建立 schema 校验、降级和兼容机制

### Version `1.3` 产品内 Skills 与学习治理

- 用能力包替代散乱规则
- 学习候选进入审核流，审批后才进入长期生效

### Version `1.4` 商业化治理与上线能力

- 补齐观测、质量指标、多租户权限、发布安全和线上回归能力

## 每版统一完成定义

每个大版本只有在下面条件全部满足后，才允许合入 `main`：

- 对应 spec、设计、任务板已更新
- 子任务全部完成并勾选
- 单测、集成测试、前端测试、E2E 全通过
- `test` 环境回归通过
- `main` 环境上线后回归通过
- 发布总结和问题复盘已入仓库

## 执行节奏

1. 先做文档和任务拆分
2. 一个小任务只解决一个问题
3. 每个小任务都先补测试或先写测试
4. 小任务完成后立刻跑对应测试
5. 小任务合回 `develop`
6. 一个大版本完成后进入 `test`
7. `test` 环境回归通过后合入 `main`
8. `main` 环境回归通过后打版本标签并写复盘
